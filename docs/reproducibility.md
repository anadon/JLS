# Reproducible Builds: declared scope and verification recipe

JLS conforms to the [Reproducible Builds](https://reproducible-builds.org)
definition — *"given the same source code, build environment and build
instructions, any party can recreate bit-by-bit identical copies of all
specified artifacts"* — for a **declared** artifact set. This document is
that declaration: it names the specified artifacts, records the build
environment a third party needs, and gives the verification recipes.
(Issue #185; determinism groundwork in #44.)

## 1. Specified artifacts (the reproducible set)

| Artifact | Release asset | Reproducible |
|---|---|---|
| Application jar | `jls-<version>.jar` | **Yes** — bit-for-bit |
| CycloneDX SBOM | `bom.json` (also built: `bom.xml`) | **Yes** — bit-for-bit |
| Maven pom | (GitHub Packages) | **Yes** — it is source |
| Native installers (`deb`, `rpm`, `msi`, `dmg`, AppImage) | `SHA256SUMS-installers-*` assets | **No** — see §5 |
| Container image | `ghcr.io/anadon/jls` | **No** — see §5 |

The reproducibility claim covers exactly the **jar and the BOM**. Nothing
in this document should be read as claiming the installers or the
container image reproduce; their integrity story is different (§5).

## 2. Build environment record: the `.buildinfo`

Every release publishes `jls-<version>.buildinfo`, generated by
`maven-artifact-plugin`'s `artifact:buildinfo` goal in the same Maven
session that built the artifacts (see `.github/workflows/release.yml`).
It is the machine-readable record in the
[JVM `.buildinfo` format](https://reproducible-builds.org/docs/jvm/) of:

- the **exact JDK** (`java.version`, `java.vendor`) and **Maven version**
  (`mvn.version`) that produced the release, and
- the **SHA-512 checksums** of every attached artifact (pom, jar, BOM).

The `.buildinfo` is the authoritative toolchain pin for a given release:
CI resolves "the latest Temurin 25 GA build" at build time (a deliberate
trade-off — the runner picks up JDK security patches automatically), so
the exact build actually used is recorded per release rather than
hard-coded here. To reproduce a past release, install the JDK build its
`.buildinfo` names (Temurin archives all past builds at
<https://adoptium.net/temurin/archive/>).

### What must match, and what does not have to

- **Must match:** the JDK build (`java.version` + `java.vendor` from the
  `.buildinfo`). Different `javac` builds may emit different (equally
  correct) class bytes; this is an environment constraint the
  Reproducible Builds definition explicitly permits recording, not a
  defect.
- **Should match:** the Maven version from the `.buildinfo`. In practice
  any Maven ≥ 3.6.3 (the enforcer floor) has produced identical output,
  but the recorded version is the supported reproduction path.
- **Verified irrelevant** (normalized by the build, enforced by CI —
  §4): build path, timezone, locale, umask, wall-clock time. Archive
  entry timestamps are pinned by `project.build.outputTimestamp` in
  `pom.xml`.

## 3. Third-party verification recipes

All commands assume the release tag `v<version>`.

### 3.1 Checksum check (integrity of what you downloaded)

```sh
gh release download v<version> --repo anadon/JLS
sha256sum -c SHA256SUMS
```

### 3.2 Provenance attestation (who built it, from what commit)

```sh
gh attestation verify jls-<version>.jar --repo anadon/JLS
```

### 3.3 Independent rebuild (the Reproducible Builds bar)

1. Install the JDK build and Maven version named in the release's
   `jls-<version>.buildinfo` (`java.version`, `java.vendor`,
   `mvn.version`).
2. Rebuild from the tagged source — any path, timezone, locale, or
   umask:

   ```sh
   git clone --branch v<version> https://github.com/anadon/JLS.git
   cd JLS
   mvn -B -DskipTests package
   ```

3. Compare against the published artifacts:

   ```sh
   sha256sum target/jls-<version>.jar target/bom.json
   # must equal the corresponding lines of the release's SHA256SUMS
   ```

   Or compare against the `.buildinfo`'s SHA-512 entries:

   ```sh
   sha512sum target/jls-<version>.jar target/bom.json
   ```

If the checksums differ while the JDK build matches, that is a bug —
please open an issue with both jars attached (`diffoscope jar1 jar2`
localizes the offending input).

### 3.4 `mvn artifact:compare` (against the Maven registry)

The jar is also published to the GitHub Packages Maven registry, so the
plugin's compare goal can diff a local rebuild against the published
binaries:

```sh
mvn -B -DskipTests package \
  org.apache.maven.plugins:maven-artifact-plugin:3.6.0:compare \
  -Dreference.repo=https://maven.pkg.github.com/anadon/JLS
```

Caveat: GitHub's Maven registry requires an access token even for public
downloads (a `settings.xml` server entry with a `read:packages` token),
so recipe 3.3 is the friction-free path; 3.4 is provided for tooling
that speaks `.buildinfo`/`compare` natively.

## 4. Continuous verification (CI gate)

The `reproducibility` job in `.github/workflows/ci.yml` is the gate, and
it checks more than same-machine determinism:

1. **Same-runner pre-filter** (#44): build twice back-to-back from
   clean; jar and BOM checksums must match.
2. **Independent perturbed rebuild** (#185): rebuild the same commit
   from a differently-named workspace path with `TZ=Pacific/Kiritimati`,
   `LC_ALL=C` (vs. the runner's `C.UTF-8`), and `umask 077` (vs. the
   default `022`), holding only the JDK build fixed; jar and BOM must
   match the reference byte-for-byte. On divergence the job fails and
   uploads both jars for `diffoscope`.

Experimental record (issue #185, P1): the perturbed rebuild was first
run 2026-07-18 at commit `49de3d6` (OpenJDK 25.0.3, Maven 3.9.11) —
jar and BOM reproduced bit-for-bit across all four perturbed axes,
supporting the hypothesis that `project.build.outputTimestamp` already
normalizes every environment-sensitive input except the JDK build
itself.

## 5. Out of scope: installers and the container image (issue #184)

The jpackage installers embed build wall-clock and signing metadata, and
the container image installs distribution packages at build time; today
they are **not** reproducible, and this document deliberately does not
claim otherwise. Their integrity model is *attestation-backed* instead:
per-file SHA-256 checksums (`SHA256SUMS-installers-*`), build-provenance
attestations (`gh attestation verify`), and — for the image — a keyless
cosign signature on the manifest digest. Making those artifacts
reproducible (apt pinning, `SOURCE_DATE_EPOCH`) is owned by issue #184;
as that lands, the table in §1 expands.
