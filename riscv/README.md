# A single-cycle RISC-V (RV32I) CPU in JLS

This directory contains a faithful **single-cycle RV32I processor built as an
actual JLS logic circuit** — real adders, multiplexers, barrel shifters,
flip-flops, an instruction ROM and a data RAM, wired together — plus the
tooling to compile RISC-V assembly into a runnable `.jls` file and to verify
the hardware against an independent reference emulator.

The CPU is not a special JLS mode or a plugin: it is an ordinary circuit made
of the elements JLS already ships (`Adder`, `Mux`, `Register`, `Memory`,
`ShiftRegister`, `Splitter`/`Binder`, `Decoder`, gates), which is exactly what
JLS is *for* — teaching computer architecture by drawing datapaths. Because
JLS simulates it as digital logic, the same circuit runs in the GUI editor and
in headless batch mode.

## Quick start

```sh
# from the repository root, build the JLS jar once:
mvn -q package -DskipTests

cd riscv

# validate the element library against the real simulator:
python3 test_primitives.py

# run the reference emulator's own self-test:
python3 riscv_ref.py

# build and run the whole CPU against the reference emulator on a directed suite:
python3 verify.py

# compile a program of your own into a runnable circuit:
python3 make_cpu.py examples/sum1to10.s -o build/sum.jls --imem 32 --dmem 16
# ...then run the printed `java -jar ... -b -t ... sum.jls` command, or open
# build/sum.jls in the JLS GUI and step the clock by hand.
```

## What works

The full **RV32I base integer instruction set**, verified instruction-by-instruction
against the reference emulator:

| Group | Instructions |
|---|---|
| Reg–imm ALU | `addi slti sltiu xori ori andi slli srli srai` |
| Reg–reg ALU | `add sub sll slt sltu xor srl sra or and` |
| Upper imm | `lui auipc` |
| Jumps | `jal jalr` (with correct link value and JALR low-bit clear) |
| Branches | `beq bne blt bge bltu bgeu` (signed and unsigned) |
| Memory (word) | `lw sw` |

Randomized differential testing (`fuzz_diff.py`) generates hundreds of random
programs and requires the hardware's final register file and data memory to
match the reference emulator exactly.

### Scope note (sub-word memory)

Loads/stores are implemented at **word granularity** (`lw`/`sw`). The data
memory is a 32-bit-word `Memory` element, so byte/half accesses (`lb lh lbu
lhu sb sh`) would need extra shift/mask/merge logic around it. The reference
emulator implements them; the hardware deliberately does not, to keep this a
*simple* core. Programs verified against the hardware use word accesses. Adding
sub-word support is a self-contained extension (a byte-lane mux on the read
path and a read-modify-write on the store path).

## Architecture

Single-cycle: every instruction completes in one clock. On each rising clock
edge the **PC** flip-flop latches the next PC and (for a writing instruction)
one register-file flip-flop latches its result — both values computed
combinationally from the *previous* cycle, the standard single-cycle timing.

```
        +----+     +--------+   instr   +---------+
  PC -->|imem|ROM->| decode |---------->| control |--- control word
   |    +----+     +--------+           |  ROM    |
   |                    |               +---------+
   |            rs1,rs2,rd, funct3           |
   |                    v                     v (RegWrite, ALUSrc, ALUctrl,
   |             +--------------+             |  ImmSel, MemRead/Write, WBSel,
   |   rs1  +----| register file|<--writeback |  Branch, JumpSel)
   |   rs2  |    | 31 FFs + 2   |     ^
   |        |    | read muxes   |     |
   |        |    +--------------+     |
   |        v                         |
   |   +---------+   +-----+          |
   |   |  imm    |   | ALU |---aluout-+--> data RAM (lw/sw)
   |   |  gen    |-->|     |          |
   |   +---------+   +-----+          |
   |                                   \--> writeback mux --> register file
   |   next-PC mux (PC+4 / branch&jal target / jalr target)
   +-------------------------------------------------------------> PC.D
```

Building blocks and the JLS elements that implement them:

- **Program counter** — a 32-bit `Register` (positive-edge flip-flop), reset
  value 0.
- **Instruction memory** — a word-addressed `Memory` ROM initialised with the
  assembled program; address is `PC[hi:2]` sliced by a `Splitter`.
- **Decode** — `Splitter`s extract `opcode`, `rd`, `funct3`, `rs1`, `rs2`,
  `funct7[5]` from the instruction word.
- **Control** — a `Memory` ROM keyed on `(opcode[6:2], funct3, funct7[5])`
  emitting a packed control word, unpacked by a `Splitter`. This is the
  classic ROM-controlled decode; its contents are generated by the same Python
  decode function that drives the reference emulator, so hardware and reference
  agree by construction.
- **Immediate generator** — the five RISC-V immediate formats (I/S/B/U/J) are
  assembled from instruction bit-ranges with `Binder`s and sign-extended with
  `Extend` (bit-31 replicated), then selected by a `Mux`.
- **Register file** — `x0` is a constant 0; `x1..x31` are 32-bit flip-flops.
  A `Decoder` on `rd` plus per-register AND gates form the write enables; each
  register has a hold/load `Mux` on its `D` input so unselected registers keep
  their value. Two 32-way `Mux`es form the `rs1`/`rs2` read ports.
- **ALU** — one `Adder` does add/subtract (B is XORed with the sub flag and
  `Cin` = sub); `AndGate`/`OrGate`/`XorGate` do the logic ops; three
  `ShiftRegister` barrel shifters do `sll`/`srl`/`sra`; `slt`/`sltu` are
  derived from the subtractor's sign and carry. A 16-way `Mux` selects the
  result by the 4-bit ALU control.
- **Branch unit** — a dedicated comparator (equality via XOR + OR-reduce, and
  a subtractor for signed/unsigned less-than) feeds an 8-way `Mux` selected by
  `funct3`; the result is ANDed with the `Branch` control bit.
- **Data memory** — a word-addressed `Memory` RAM. Its write-enable is gated
  so a store commits only in the clock-low phase, after the datapath has
  settled, preventing spurious writes to transient addresses.

## Files

| File | Purpose |
|---|---|
| `jlsbuild.py` | Netlist compiler: emits `.jls` text; one factory per element, exact save-format and put names. Fan-out becomes a single star net. |
| `jlsrun.py` | Runs a circuit in JLS batch mode and parses the watched-element output. |
| `build_cpu.py` | Constructs the RV32I datapath and the decode ROM. |
| `riscv_ref.py` | Independent RV32I assembler + emulator (the golden oracle). |
| `make_cpu.py` | CLI: assembly `.s` → runnable `.jls` + clock vector + expected state. |
| `verify.py` | Directed end-to-end suite: hardware vs reference. |
| `fuzz_diff.py` | Randomized differential tester (parallel). |
| `test_primitives.py` | Validates every element emitter against the real simulator. |
| `examples/*.s` | Sample programs. |

## How verification works

JLS batch mode prints the final value of every *watched* `Register` and
`Memory`. The CPU marks all 31 architectural registers, the PC, and the data
RAM as watched, so a batch run reports the complete architectural state. The
clock is an input pin driven by a `-t` test vector that supplies exactly as
many rising edges as the reference emulator took to finish, then holds — so the
hardware halts in the same architectural state the emulator predicts, and the
two are compared register-by-register and word-by-word.

Nothing here modifies JLS itself: the CPU and all tooling sit on top of the
unmodified simulator and its documented file/batch formats.
